# Миграция на @module-federation/vite с поддержкой HMR

## Цель
Получить полноценный HMR на обеих сторонах (Host + Remote) для всех микрофронтендов.

## Контекст
`@originjs/vite-plugin-federation` генерирует `remoteEntry.js` только при `vite build`. В dev режиме Remote требует пересборки. Переходим на `@module-federation/vite` с нативной поддержкой HMR.

**Пример для Hub (Host):**
```tsx

const PulseDevices = defineAsyncComponent(() => import('pulse/Devices'));
const PulseMetrics = defineAsyncComponent(() => import('pulse/Metrics'));
const PulseAlerts = defineAsyncComponent(() => import('pulse/Alerts'));
const PulseSettings = defineAsyncComponent(() => import('pulse/Settings'));

// В Routes:
<Route path="/pulse/dashboard" element={<PulseDashboard />} />
<Route path="/pulse/devices" element={<PulseDevices />} />
<Route path="/pulse/metrics" element={<PulseMetrics />} />
<Route path="/pulse/alerts" element={<PulseAlerts />} />
<Route path="/pulse/settings" element={<PulseSettings />} />
```

**Аналогично для service и control** - свои модули.

## Риски и решения

| Риск | Вероятность | Решение |
|------|-------------|---------|
| `defineAsyncComponent` не работает с MF в dev | Средняя | Проверить на шаге 4, возможно нужен обычный `lazy()` |
| @module-federation/vite несовместим с React 18 | Низкая | Использовать `React.lazy` или проверить совместимость |
| Порты конфликтуют в dev | Низкая | Проверить что каждый сервис на своём порту |
