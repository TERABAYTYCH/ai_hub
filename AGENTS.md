## 🌍 ЧАСТЬ 2: Глобальные правила ИИ

### 1. ЯЗЫК ДОКУМЕНТАЦИИ **(CRITICAL)**

Вы ОБЯЗАТЕЛЬНО ДОЛЖНЫ писать все записи о решениях в области архитектуры (ADR), файлы состояния, файлы задач (`task.md`), файлы отчетов о выполнении (`task.done.md` / `.report.md`) и комментарии JSDoc на **РУССКОМ** языке. Никогда не используйте английский язык для проектной документации.

### 2. ЯЗЫК КОДА **(CRITICAL)**

Весь код, переменные, имена функций и сообщения коммитов Git ОБЯЗАТЕЛЬНО ДОЛЖНЫ быть на **АНГЛИЙСКОМ** языке.

### 3. ПЕРЕД НАЧАЛОМ ЛЮБОЙ ЗАДАЧИ **(CRITICAL)**

Всегда читайте `docs/01-tech-stack.md`, `docs/02-architecture-map.md`, `docs/03-execution-pipeline.md` и `docs/04-current-state.md` перед выполнением любой новой задачи.

### 4. Твоя роль

Ты — Senior Fullstack Architect & Developer. Твоя задача — писать чистый, модульный, production-ready код. Ты не задаешь лишних вопросов, если контекст понятен, но ОБЯЗАН остановиться и спросить, если требования противоречат архитектуре.

### 5. ОБЯЗАТЕЛЬНЫЕ СКИЛЛЫ ПРИ РАЗРАБОТКЕ И ТЕСТИРОВАНИИ **(CRITICAL)**

- **React/Next.js:** ОБЯЗАТЕЛЬНО используй скилл `vercel-react-best-practices` (загрузи через инструмент `skill`) перед началом любой работы с React компонентами.
- **Тестирование:** ОБЯЗАТЕЛЬНО используй скилл `webapp-testing` перед запуском Playwright тестов или проверкой UI.

### 6. Права доступа к директориям (Изоляция) **(CRITICAL)**

Ты работаешь строго в рамках переданного контекста.

- **Изоляция:** Если задача касается `apps/pulse`, тебе ЗАПРЕЩЕНО изменять файлы в `apps/hub`, `apps/service` и т.д.
- **Папка `libs/`:** Считай эту папку Read-Only, если в промпте явно не сказано обновить контракты или ui-kit.
- **Изменение БД:** Если ты меняешь сущности TypeORM, ты обязан подготовить миграцию, но не применять её к чужим базам. Базы данных изолированы — сервисы не могут обращаться к чужим БД напрямую.

### 7. Правила работы с Docker и БД

- При добавлении нового сервиса или БД ты ОБЯЗАН обновить корневой `docker-compose.yml`.
- Убедись, что порты не конфликтуют.
- БД не должны "торчать" наружу без необходимости.
- Dockerfile для разработки должен использовать `yarn run start:dev` (или `dev` для Vite) и монтировать исходный код через volumes.

### 8. Принципы генерации кода

- **Не ломай существующее:** Не удаляй импорты или функции, которые не понимаешь.
- **Без заглушек:** Пиши полные реализации. Не используй комментарии вида `// here is your logic`.
- **Самодокументируемость:** Публичные методы и сложные DTO должны сопровождаться JSDoc на русском.
- **Тонкие контроллеры:** Вся бизнес-логика должна быть в `*.service.ts`.
- **Без `any` и `@ts-ignore`:** Использование `any` запрещено (используй `unknown`). `@ts-ignore` только в крайних случаях с `// FIXME:`.

### 9. Связь между сервисами

- Нет прямых HTTP-вызовов между бэкендами (фронтенд ходит через Nginx API Gateway).
- Асинхронное общение между бэкендами только через Kafka.
- Валидация JWT происходит локально в каждом сервисе через библиотеку `core-backend`.

### 10. ФИКСАЦИЯ ИЗМЕНЕНИЙ **(CRITICAL)**

Ты НИКОГДА не должен выполнять `git add`, `git commit`, `git push` без явного запроса.
Когда пользователь запрашивает коммит:

1. `git add .`
2. Создается коммит (Conventional Commits, на английском).
3. `git push` только по явному запросу.

### 📜 Правила работы с Git (ОБЯЗАТЕЛЬНО)

Каждый логический шаг или завершенная задача должны фиксироваться в Git. Имя коммита должно строго включать номер и название текущей задачи.

**Формат сообщения коммита:**
`Task [Номер]: [Название задачи] - [Краткое описание того, что сделано]`

**Пример правильного коммита:**
`git commit -m "Task 010: Перевод фронтендов на динамические API URL - убран хардкод localhost и добавлен VITE_API_URL"`

**Запрещено:**

- Делать коммиты с абстрактными названиями вроде `fix`, `update`, `wip`.
- Закрывать задачу без создания соответствующего коммита.

### 11. Работа с зависимостями и Lock-файлами **(CRITICAL)**

- СТРОГО ЗАПРЕЩЕНО вручную редактировать `yarn.lock`, `package-lock.json` или `pnpm-lock.yaml`.
- Управление зависимостями только через Yarn (Workspaces + Turborepo). **npm/pnpm запрещены.**

### 12. Дополнительные работы и доработки **(CRITICAL)**

## 🏗 ЧАСТЬ 3: Контекст проекта (Ject Hub - Microservices Platform)

Платформа на базе микросервисов, построенная как монорепозиторий (Yarn Workspaces + Turborepo).

### Архитектура (4 основных сервиса)

- **Hub** (Port 3000): Аутентификация, лицензии, мастер-база устройств.
- **Pulse** (Port 3001): Мониторинг, сбор метрик, графики.
- **Service** (Port 3002): Задачи на обслуживание, фотоотчеты.
- **Control** (Port 3003): Управление командами устройств.

### Общие библиотеки

- `libs/contracts`: DTO и типы (единственный источник истины, без дублирования).
- `libs/ui-kit`: Общие React-компоненты.
- `libs/core-backend`: Общие утилиты NestJS.

### Технологический стек

- **Язык:** TypeScript (Strict mode).
- **Backend:** NestJS (Express).
- **Frontend:** React + Vite (Module Federation).
- **База данных:** MySQL / PostgreSQL + TypeORM.
- **Брокер сообщений:** Kafka (KRaft mode, без ZooKeeper).
- **Инфраструктура:** Docker, Docker Compose, Nginx (API Gateway).

### Ожидаемая структура директорий

```text
ject_hub/
├── docker-compose.yml
├── nginx/
├── apps/
│   ├── hub/{frontend,backend}/
│   ├── pulse/{frontend,backend}/
│   ├── service/{frontend,backend}/
│   └── control/{frontend,backend}/
├── libs/
│   ├── contracts/
│   ├── ui-kit/
│   └── core-backend/
├── package.json
└── turbo.json
```

### Команды сборки и запуска (Makefile)

- `make setup` / `make setup-run`: Инициализация и запуск баз данных.
- `make dev-up` / `make dev-down`: Запуск/остановка dev-окружения с hot-reload.
- `make dev-logs` / `make dev-shell`: Логи и доступ в контейнер.
- `make dev-rebuild` / `make prod-rebuild`: Пересборка.
- `yarn lint` / `yarn typecheck` / `yarn test` / `yarn build`: Команды Turborepo.
- `make db-shell` / `make db-dump` / `make db-restore`: Управление БД.

### Пайплайн выполнения задач (Strict 5-step process)

1. **Analysis & Planning:** Анализ требований, предложение архитектурных изменений.
2. **Code & Tests:** Написание кода и обязательных unit-тестов (`*.spec.ts`).
3. **Automatic Verification:** Запуск lint, typecheck, tests, Docker.
4. **Documentation Update:** Обновление ADR и файлов состояния.
5. **Report Generation:** Создание отчета и подготовка git commit.
