# Карта архитектуры и взаимодействия

## 1. Структура монорепозитория

repository/
├── docker-compose.yml # Главный файл оркестрации (БД, Kafka, Nginx, сервисы)
├── nginx/ # Конфиги API Gateway
├── apps/
│ ├── hub/ # Ядро (Порт 3000). Auth, лицензии, мастер-база устройств.
│ ├── pulse/ # Мониторинг (Порт 3001). Сбор метрик, графики.
│ ├── service/ # Обслуживание (Порт 3002). Задачи, фотоотчеты.
│ └── control/ # Управление (Порт 3003). Отправка команд на устройства.
├── libs/
│ ├── contracts/ # Единый источник истины для DTO и типов.
│ ├── ui-kit/ # Общие React-компоненты.
│ └── core-backend/ # Общие модули NestJS (Auth Guards, Kafka Client).

## 2. Правила общения сервисов и Docker-сети

- **Внутренняя сеть (Docker internal network):** Бэкенды общаются с БД и брокером сообщений по внутренним DNS-именам контейнеров (например, `mysql-hub:3306`, `kafka:9092`).
- **Синхронно (REST API):** Фронтенд общается со своим бэкендом ТОЛЬКО через API Gateway (Nginx разруливает роуты `/hub/api`, `/pulse/api` на нужные контейнеры).
- **Асинхронно (Kafka):** Бэкенды общаются между собой ТОЛЬКО через события.
- **Авторизация:** Hub выдает JWT. Остальные сервисы валидируют его локально через библиотеку из `core-backend` (без запросов в Hub).

## 3. Микрофронтенды (Module Federation)

- Любой модуль может быть как Host (основным окном), так и Remote (встраиваемым компонентом).
- Навигация между модулями должна быть бесшовной, без перезагрузки страницы.
