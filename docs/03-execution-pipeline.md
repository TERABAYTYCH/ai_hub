# Пайплайн выполнения задач (Strict Workflow)

При получении любого запроса от пользователя, ты ОБЯЗАН выполнить следующие шаги в строгом порядке.
**Распределение ролей:** Пользователь выполняет вручную ТОЛЬКО действия с Git. Запуск скриптов, тестов, линтеров и поднятие Docker-контейнеров — твоя прямая обязанность через доступные тебе CLI-инструменты.

## Шаг 0: Навигация по задачам (Доменная структура)

Все задачи проекта строго разделены по доменам и хранятся в `docs/tasks/`:

- `0_infrastructure/` — корневые конфиги, CI/CD, Docker.
- `1_hub/` — ядро системы (frontend + backend).
- `2_pulse/`, `3_service/`, `4_control/` — остальные микросервисы.
  Всегда проверяй путь к файлу задачи, над которой работаешь, и сохраняй контекст соответствующего домена. Файлы-отчеты (`.report.[HHMM].md`) всегда сохраняются в той же папке, что и сама задача.

## Шаг 1: Анализ и планирование

- Прочитай переданный контекст и задачу из соответствующей доменной папки.
- Если задача затрагивает изменение архитектуры, добавление новых сущностей БД (TypeORM) или изменение инфраструктуры (Docker), сначала предложи решение. Если задача рядовая (CRUD, компонент) — переходи к Шагу 2.

## Шаг 2: Написание кода и Тестов (Атомарность)

- Напиши или измени код строго в рамках запрошенной фичи.
- **Тестирование (ОБЯЗАТЕЛЬНО):** Для любой новой или измененной бизнес-логики ты ОБЯЗАН написать Unit-тесты в файлах `*.spec.ts`. Код без тестов считается невыполненным.
- Если для фичи нужны новые DTO, сначала создай их в `libs/contracts`, и только потом используй в микросервисе.
- Если добавлены новые зависимости или изменены порты, самостоятельно обнови `package.json` и `docker-compose.yml`.

## Шаг 3: Автоматическая проверка (ОБЯЗАТЕЛЬНО)

После генерации кода ты ОБЯЗАН самостоятельно выполнить проверки через свой терминал. Если любая из команд падает с ошибкой, ты должен проанализировать лог и исправить код, прежде чем переходить к Шагу 4.

1. **Линтер:** Запусти `yarn lint` для измененных файлов. Исправь все ошибки.
2. **Типизация:** Запусти `yarn typecheck`.
3. **Тесты:** Запусти `yarn test` и добейся 100% прохождения.
4. **БД:** _Если менялись сущности TypeORM_, самостоятельно сгенерируй миграции.
5. **Инфраструктура:** _Если менялся Docker_, выполни `docker compose up -d --build` и убедись, что контейнеры поднялись успешно.

## Шаг 4: Обновление документации (State и ADR)

- **ADR:** Если принято архитектурное решение, создай/обнови файл в `docs/adr/`.
- **State:** Обнови файл `docs/04-current-state.md`. Запиши туда: что было сделано только что, и какой логический шаг должен быть следующим.

## Шаг 5: Создание файла-отчета и передача управления (Ready for Git)

Когда код написан, проверен тобой в терминале и документация обновлена, зафиксируй результат.

1. **Создание отчета (`.report.[HHMM].md`):** Создай файл отчета **в той же директории, где находится исходный файл задачи**. Обязательный формат имени файла: `[имя-задачи].report.[HHMM].md`. Например: `001-init.report.1200.md`. Использование формата без даты **ЗАПРЕЩЕНО**.
   - Дату и время определяй по дате выполнения задачи (указана внутри отчета) или по времени создания файла.
2. **Вывод в чат:** Выведи сообщение по шаблону:
   - **Что сделано:** (Кратко подтверди завершение и дай ссылку на `.report.[дата].md`).
   - **Результаты проверок:** (Подтверди успешный прогон typecheck, lint, тестов).
   - **Сообщение для коммита:** Предложи готовое сообщение для `git commit`.
   - **Ожидание:** Сообщи, что ждешь, пока пользователь сделает коммит вручную.

## 🐳 Docker Dev-режим (Hot-Reload)

При разработке с использованием Docker-контейнеров для фронтенд-приложений (Hub, Pulse) используется специальный dev-режим.

### СтартDev-процесс разработки

| **Шаг** | **Было (Старый пайплайн)** | **Стало (Новый пайплайн)** |
|----------|----------------------------|----------------------------|
| **При изменении кода** | Остановка → `docker compose build` → Запуск | Код обновляется мгновенно (Vite HMR) |
| **При изменении зависимостей** | Полная пересборка образа | `docker compose restart <service>` |
| **Доступ к remoteEntry** | Из статического билда Nginx/Serve | Перехватывается Vite и отдается из `dist` |

**Важно:** Вместо пересборки проекта теперь достаточно просто перезапустить контейнеры (`docker compose restart <service>`), так как установка пакетов и сборка `remoteEntry.js` происходят при старте контейнера автоматически.

### Как работает Hot-Reload

1. При старте контейнера выполняется `yarn dev:docker`
2. Команда `yarn install && yarn build && vite --host` гарантирует:
   - Актуальные зависимости
   - Собранный `remoteEntry.js` в `dist/assets/`
   - Запущенный dev-сервер Vite
3. Vite плагин `remoteEntryPlugin` перехватывает запросы к `/assets/remoteEntry.js` и отдает файл из `dist/assets/`
4. Изменения кода на хосте мгновенно применяются через volumes + Vite HMR

### Команды управления

```bash
# Полная пересборка (редко нужно)
docker compose up -d --build

# Перезапуск (при изменении зависимостей)
docker compose restart hub-frontend
docker compose restart pulse-frontend

# Только перезапуск без build
docker compose up -d
```
