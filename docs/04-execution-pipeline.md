# Пайплайн выполнения задач (Strict Workflow)

При получении любого запроса от пользователя, ты ОБЯЗАН выполнить следующие шаги в строгом порядке. Пользователь выполняет все действия с Git и запуском контейнеров вручную, твоя задача — подготовить для этого идеальный код и конфигурации.

## Шаг 1: Анализ и планирование

- Прочитай переданный контекст и задачу.
- Если задача затрагивает изменение архитектуры, добавление новых сущностей БД (TypeORM) или изменение инфраструктуры (Docker), сначала предложи решение. Если задача рядовая (CRUD, компонент) — переходи к Шагу 2.

## Шаг 2: Написание кода (Атомарность)

- Напиши или измени код строго в рамках запрошенной фичи.
- Если для фичи нужны новые DTO, сначала создай их в `libs/contracts`, и только потом используй в микросервисе.
- Если добавлены новые зависимости или изменены порты, обнови `package.json` и `docker-compose.yml`.

## Шаг 3: Автоматическая проверка (ОБЯЗАТЕЛЬНО)

После генерации кода ты должен выполнить проверки через CLI (если среда позволяет) или предоставить пользователю команды для проверки:

1. Запусти `yarn lint` для измененных файлов. Исправь все найденные ошибки.
2. Запусти `yarn typecheck` (или `yarn workspace <имя_модуля> typecheck`) в рабочей области.
3. Убедись, что нет неиспользуемых импортов и ошибок типизации.
4. _Если менялись сущности TypeORM:_ Напомни пользователю сгенерировать и применить миграции.
5. _Если менялся Dockerfile или docker-compose.yml:_ Напомни пользователю выполнить `docker compose up -d --build`.

## Шаг 4: Обновление документации (State и ADR)

- **ADR:** Если в ходе задачи было принято архитектурное решение (новая таблица, новый паттерн, изменение сети Docker), создай или обнови файл в `docs/adr/`.
- **State:** Обнови файл `docs/05-current-state.md` (создай его, если нет). Запиши туда: что было сделано только что, и какой логический шаг должен быть следующим.

## Шаг 5: Отчет и передача управления (Ready for Git)

Когда код написан и проверен, выведи сообщение по шаблону:

1. **Что сделано:** (Краткий список измененных файлов).
2. **Результаты проверок:** (Успешно ли прошел typecheck).
3. **Сообщение для коммита:** Предложи пользователю готовое сообщение для `git commit` в формате Conventional Commits (например, `feat(pulse): add metrics entity and update docker-compose`).
4. **Ожидание:** Сообщи, что ждешь, пока пользователь сделает коммит вручную, чтобы перейти к следующему шагу.
