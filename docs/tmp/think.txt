Разработка микросервисного приложения
Стек Typescript + Vite-ReactTS + NestJS 

Первоначальная задумка
	Микросервис 1 - Сервис мониторинга, сбор данных с различных устройств, агрегирование графиков. Устройства те же что и в микросервисах 2 и 3. 
	Микросервис 2 - Сервис ведения отчетности обслуживания различных устройств. Устройства те же что и в микросервисах 1 и 3. Задачи на обслуживание формируются исходя из статистики использования в приложении 1
	Микросервис 3 - Сервис управления различными устройствами. Устройства те же что и в микросервисах 1 и 2. 
	Микросервис-основа 0 - "Хаб" для всех микросервисов. Общая база устройств и остальных общих сущностей. Место хранения информации о купленных лицензиях

Названия пусть будут такие
	Микросервис 1 - Pulse
	Микросервис 2 - Service
	Микросервис 3 - Control
	Микросервис 0 - Hub

Описание приложения
	Приложение подразумевает продукт который можно продавать частями. 
	Возможно использование каждого микросервиса отдельно. 
	Возможна "допокупка" другого микросервиса. 
	У всех микросервисов есть свой фронтенд, доступный по ссылке. При открытии ссылки микросервис будет "основным" и иметь в панели остальные доступные микросервисы как пункты меню. Например в Hub боковая панель навигации при наличии всех лицензий имеет пункты Pulse, Service, Control, Настройки. Каждый пункт имеет подпункты, которые открывают панели микросервиса (например Мониторинг - графики, Обслуживание - отчеты и тд) без перехода в режим "Микросервис - основа". А при открытии /pulse (мониторинг), уже режим "Микросервис - основа", боковая панель имеет свои основные пункты (устройства, графики, уведомления) и, при наличии лицензий, пункты Service, Control. Доступ в HUB из под микросервиса нужен только когда куплено больше одного микросервиса. И такая связь нужна внутри всех микросервисов.

Фронтенды
	Hub (ссылка domain.com/hub) - Панель для управления/переключения между микросервисами, управление лицензией и тд
	Pulse (ссылка domain.com/pulse) - Панель мониторинга устройств. Добавление устройств, Просмотр графиков, статистики, состояния каждого устройства
	Service (ссылка domain.com/service) - Панель ведения отчетности. Добавление устройств, Просмотр устройств, которым необходимо обслуживание, проведение работ с добавлением фотоотчетов, просмотр отчетов о проведенных работах
	Control (ссылка domain.com/control) - Панель управления устройствами. Добавление устройств, управление устройствами, объединение устройств в группы

Бекенды
	Hub (ссылка domain.com/hub/api) - Общая база
	Pulse (ссылка domain.com/pulse/api) - 
	Service (ссылка domain.com/service/api) - 
	Control (ссылка domain.com/control/api) - 

Примерная архитектура
	repository/
	├── apps/
	│   ├── hub/                  # 🏢 Ядро системы
	│   │   ├── frontend/         # React SPA (Host)
	│   │   ├── backend/          # NestJS (Главный API)
	|   |   └── shared/           # TS-интерфейсы только для модуля Hub (DTO)
	│   │
	│   ├── pulse/                # 📈 Модуль мониторинга
	│   │   ├── frontend/         # React SPA (Remote)
	│   │   ├── backend/          # NestJS (Обработка метрик)
	|   |   └── shared/           # TS-интерфейсы только для модуля Pulse (DTO)
	│   │
	│   ├── service/              # 🛠 Модуль обслуживания
	│   │   ├── frontend/         # React SPA (Remote)
	│   │   ├── backend/          # NestJS (Отчеты)
	|   |   └── shared/           # TS-интерфейсы только для модуля Service (DTO)
	│   │
	│   └── control/              # 🛠 Модуль управления
	│       ├── frontend/         # React SPA (Remote)
	│       ├── backend/          # NestJS (Устройства, группы, пульты)
	|		└── shared/           # TS-интерфейсы только для модуля Pulse (DTO)
	│
	├── libs/                     # 🧩 Общий код (Shared)
	│   ├── contracts/            # Общие типы и события Kafka
	│   ├── ui-kit/               # React-компоненты
	│   └── core-backend/         # Общие утилиты NestJS (Kafka, Auth)

Конфиги, сервисы, библиотеки

	Папка libs должна легко импортироваться в любой микросервис
	Общение между бекендами через сервис доставки (kafka или подобное)

Задача
	Описать архитектуру микросервисов, рассказать как лучше наладить общение между сервисами, как лучше наладить общение между бекендами-фронтендами, как лучше сделать бесшовным открытие одного фронтенда внутри другого
