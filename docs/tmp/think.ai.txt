Разработка микросервисного приложения
Стек: Typescript + Vite-ReactTS + NestJS 

Первоначальная задумка
- Микросервис 1 (Pulse): Сервис мониторинга, сбор данных с различных устройств, агрегирование графиков. Устройства те же, что и в микросервисах 2 и 3. 
- Микросервис 2 (Service): Сервис ведения отчетности обслуживания различных устройств. Устройства те же, что и в микросервисах 1 и 3. Задачи на обслуживание формируются исходя из статистики использования в приложении 1.
- Микросервис 3 (Control): Сервис управления различными устройствами. Устройства те же, что и в микросервисах 1 и 2. 
- Микросервис-основа 0 (Hub): "Хаб" для всех микросервисов. Общая база устройств и остальных общих сущностей. Место хранения информации о купленных лицензиях.

Описание продукта и бизнес-логики
Приложение подразумевает продукт, который можно продавать частями (модулями). 
- Возможно использование каждого микросервиса отдельно. 
- Возможна "допокупка" другого микросервиса. 
- У всех микросервисов есть свой фронтенд, доступный по ссылке. При открытии ссылки микросервис становится "основным" (Host) и имеет в панели остальные доступные (купленные) микросервисы как пункты меню. 
- Пример навигации: В Hub боковая панель при наличии всех лицензий имеет пункты Pulse, Service, Control, Настройки. Каждый пункт имеет подпункты, которые открывают панели микросервиса (например, Мониторинг -> Графики, Обслуживание -> Отчеты) без перехода в режим "Микросервис-основа". 
- А при прямом открытии /pulse (мониторинг), уже он выступает как "Микросервис-основа". Боковая панель имеет свои основные пункты (устройства, графики, уведомления) и, при наличии лицензий, пункты Service, Control. 
- Доступ в HUB из-под микросервиса нужен только когда куплено больше одного модуля. Такая бесшовная связь нужна внутри всех микросервисов.

Фронтенды (Микрофронтенды на Vite Module Federation)
- Hub (ссылка domain.com/hub): Панель для управления/переключения между микросервисами, управление лицензиями, пользователями.
- Pulse (ссылка domain.com/pulse): Панель мониторинга устройств. Добавление устройств, просмотр графиков, статистики, состояния каждого устройства.
- Service (ссылка domain.com/service): Панель ведения отчетности. Добавление устройств, просмотр устройств, которым необходимо обслуживание, проведение работ с добавлением фотоотчетов, просмотр истории работ.
- Control (ссылка domain.com/control): Панель управления устройствами. Добавление устройств, прямое управление, объединение устройств в группы.

Бэкенды (API Gateway / Nginx роутинг)
- Hub API (ссылка domain.com/hub/api): Управление лицензиями, пользователями, мастер-база устройств. Выступает провайдером авторизации (выдает JWT).
- Pulse API (ссылка domain.com/pulse/api): Обработка метрик, телеметрии, графиков.
- Service API (ссылка domain.com/service/api): Логика задач на обслуживание, хранение отчетов.
- Control API (ссылка domain.com/control/api): Логика отправки команд на устройства, группировки.

Конфиги, сервисы, инфраструктура
- Монорепозиторий: Turborepo + Yarn Workspaces.
- Контейнеризация: Обязательное использование Docker и Docker Compose. Вся инфраструктура (каждый микросервис, базы данных, брокер сообщений, API Gateway) должна подниматься и связываться через docker-compose.yml для локальной разработки и развертывания.
- Общие библиотеки: Папка libs импортируется в любой микросервис как локальный пакет (без сложной магии).
- Базы данных: У каждого микросервиса своя изолированная БД (PostgreSQL). Hub является "источником правды" для общих сущностей (Устройства, Пользователи) и реплицирует их в другие сервисы.
- Общение бэкендов: Асинхронное, через брокер сообщений (Kafka или RabbitMQ).
- Авторизация: Единая (SSO). Hub выдает JWT-токены, остальные сервисы проверяют их автономно через общую логику из core-backend.

Задача
Описать детальную архитектуру взаимодействия микросервисов. В частности:
1. Как лучше наладить общение и синхронизацию данных между бэкендами (особенно синхронизацию базы устройств из Hub в остальные сервисы).
2. Как лучше спроектировать API Gateway и взаимодействие между фронтендом и бэкендами.
3. Как технически реализовать бесшовное открытие одного фронтенда внутри другого (Module Federation), учитывая, что любой модуль может выступать как Host-приложением, так и Remote-компонентом в зависимости от точки входа.
4. Как правильно организовать конфигурацию Docker Compose для удобной локальной разработки монорепозитория (проброс портов, hot-reload, сети между контейнерами).
