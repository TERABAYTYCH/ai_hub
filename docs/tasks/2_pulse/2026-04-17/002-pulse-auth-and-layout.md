# Задача 002: Регистрация, авторизация и базовый Layout в Pulse

## Контекст
Микросервису Pulse требуется собственный функционал регистрации и авторизации, а также базовая оболочка интерфейса (Layout), идентичная основному приложению Hub. 
Поскольку единой точкой хранения пользователей является база данных **Hub**, Hub Backend будет выполнять роль Identity Provider (выпускать JWT-токены), а Pulse Backend — роль Resource Server (проверять токены). 

**ВНИМАНИЕ:** При разработке React-компонентов обязательно использовать скилл `vercel-react-best-practices` для обеспечения оптимальной производительности и правильного управления состоянием.

## Требования и шаги реализации

### 1. Базовый Layout для Pulse Frontend (Standalone)
- Создать компонент `AppLayout` в `apps/pulse/frontend`, визуально идентичный Layout из Hub.
- **Header:** Добавить верхнюю панель (логотип/название "Pulse", профиль пользователя).
- **Sidebar (Боковое меню):** Добавить боковое меню. Пока сделать в нем заглушки (например: "Дашборд", "Метрики", "Алерты", "Настройки").
- Убедиться, что при изолированном запуске Pulse (standalone) разработчик видит полноценный интерфейс. 
- *Примечание:* При экспорте через Module Federation в Hub (компонент `PulseDashboard`) экспортироваться должна только контентная часть, без дублирования сайдбара и хедера Хаба.

### 2. Подготовка контрактов (libs/contracts)
- Вынести DTO для регистрации (`RegisterDto`) и авторизации (`LoginDto`), а также интерфейс ответа с токеном (`AuthResponse`) в общую библиотеку `libs/contracts`, чтобы оба бэкенда и фронтенда могли их использовать.

### 3. Настройка Hub Backend (Identity Provider)
- Если еще не реализовано, создать `AuthModule` в `apps/hub/backend`.
- Реализовать эндпоинты:
  - `POST /api/hub/auth/register` — создание пользователя в БД Хаба и возврат JWT.
  - `POST /api/hub/auth/login` — проверка учетных данных и возврат JWT.
- Использовать общий секретный ключ для JWT (например, задать `JWT_SECRET` в `.env`).

### 4. Настройка Pulse Backend (Resource Server)
- Установить зависимости для работы с JWT (`@nestjs/jwt`, `@nestjs/passport`, `passport-jwt`).
- Создать `JwtAuthGuard` и `JwtStrategy`.
- Настроить `JwtStrategy` на использование того же `JWT_SECRET`, что и в Hub Backend. Это позволит Pulse Backend валидировать токены без прямого обращения к БД Хаба.
- Защитить эндпоинты Pulse (например, `GET /api/pulse/health` или новые) с помощью `@UseGuards(JwtAuthGuard)`.

### 5. Реализация UI авторизации в Pulse Frontend
- Создать страницы/компоненты `RegisterPage.tsx` и `LoginPage.tsx` в `apps/pulse/frontend`.
- Настроить отправку данных форм на эндпоинты Хаба (`/api/hub/auth/register` и `/api/hub/auth/login`).
- При успешной авторизации сохранять полученный JWT-токен (в `localStorage` или cookie).
- Настроить Axios-интерцептор (или fetch) в Pulse Frontend, чтобы ко всем запросам на `/api/pulse/*` автоматически прикреплялся заголовок `Authorization: Bearer <token>`.
- Добавить логику выхода (Logout) в Header (очистка токена и редирект на Login).

### 6. Инфраструктура (Nginx)
- Убедиться, что Nginx корректно маршрутизирует запросы:
  - `/api/hub/*` -> `hub-backend`
  - `/api/pulse/*` -> `pulse-backend`

## Критерии приемки (DoD)
1. При локальном запуске Pulse Frontend отображается полноценный Layout (Header + Sidebar с заглушками).
2. Пользователь может зарегистрироваться и авторизоваться через формы в интерфейсе Pulse. Запись о пользователе появляется в базе данных Hub.
3. Запросы от фронтенда Pulse к защищенным эндпоинтам бэкенда Pulse успешно проходят аутентификацию (Pulse Backend принимает токен, выпущенный Hub Backend).
4. При попытке обратиться к защищенному эндпоинту Pulse Backend без токена возвращается ошибка `401 Unauthorized`.
5. Экспортируемый через Module Federation `PulseDashboard` корректно встраивается в Hub без двойного сайдбара/хедера.
6. Код React-компонентов написан с учетом `vercel-react-best-practices` (отсутствуют лишние ререндеры, правильная обработка состояний).
