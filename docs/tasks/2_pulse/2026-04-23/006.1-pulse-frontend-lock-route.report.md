## Доработки

**Доработка #1 - 2026-04-23 — Глобальная блокировка Pulse**

Если `microservices.pulse === false`, все routes редиректятся на `/lock`.

Добавлен хук `usePulseAccess()` который проверяет `user?.microservices['pulse']`. При загрузке:
- Если `isLoading` — показывает Loading...
- Если `isPulseLocked` — все routes редиректятся на `/lock`

**Файлы:** `apps/pulse/frontend/src/App.tsx`

**Доработка #2 - 2026-04-23 — Исправлено нарушение Rules of Hooks**

Исходная реализация вызывала `usePulseAccess()` → `useAuth()` и `useDynamicRoutesConfig()` → `useAuth()` в разном порядке при условном рендеринге. Исправлено: `useAuth()` вызывается только в `App()`, `microservicesAccess` передаётся как параметр.

**Доработка #3 - 2026-04-23 — useDynamicRoutesConfig проверяет isLocked других сервисов**

При загрузке remote модулей других сервисов — если `microservicesAccess[manifest.serviceId] === false`, добавляется route `/{serviceId}/*` → `/{serviceId}/lock`.

**Файлы:** `apps/pulse/frontend/src/App.tsx`

**Доработка #4 - 2026-04-23 — Исправлено нарушение Rules of Hooks**

Проблема была в том, что условный возврат с `useRoutes()` при `isPulseLocked` нарушал порядок хуков.

Решение:
- Разделены компоненты: `LoadingScreen`, `AppRoutes`, `App`
- `App()` всегда вызывает `useAuth()` первым
- Если `isLoading` — возвращает `LoadingScreen` (без хуков)
- Если не `isLoading` — возвращает `AppRoutes()`
- `AppRoutes()` проверяет `isPulseLocked` и редиректит на `/lock` при необходимости

**Файлы:** `apps/pulse/frontend/src/App.tsx`

**Доработка #5 - 2026-04-23 — Pulse Layout показывает только "Locked" когда заблокирован**

Если `microservices.pulse === false`:
- В Layout показывается только один пункт меню "Locked" с иконкой `bi bi-lock`
- Обычные пункты (Dashboard, Devices, etc.) не показываются

Pulse Layout теперь использует `useAuth()` и проверяет `microservicesAccess['pulse']`.

**Файлы:** `apps/pulse/frontend/src/components/layout/Layout.tsx`

**Доработка #6 - 2026-04-23 — Скрытие Hub Settings когда Pulse заблокирован**

При `isPulseLocked = true`:
- `hubSettingsItem` передаётся как `undefined` — Hub Settings скрыт
- Остальные remote сервисы (Service) могут появляться из manifests, но пользователь заблокирован на /lock и не может навигироваться

**Файлы:** `apps/pulse/frontend/src/components/layout/Layout.tsx`

**Доработка #7 - 2026-04-23 — Исключение Service из сайдбара когда Pulse заблокирован**

При `isPulseLocked = true`:
- `excludeServices` = `['pulse', 'service']` — Service не показывается в сайдбаре
- Hub Settings скрыт

**Файлы:** `apps/pulse/frontend/src/components/layout/Layout.tsx`

**Доработка #8 - 2026-04-23 — Редирект /lock и неизвестных роутов на /dashboard**

- `/lock` → `/dashboard` (когда Pulse не заблокирован)
- `/*` → `/dashboard` (catch-all для неизвестных роутов)

**Файлы:** `apps/pulse/frontend/src/App.tsx`

**Доработка #8 (исправление) - 2026-04-23 — Убран лишний /lock redirect**

Удалён отдельный route `/lock → /dashboard` — catch-all `/*` перехватывает все неизвестные пути включая `/lock`.

**Файлы:** `apps/pulse/frontend/src/App.tsx`

**Доработка #9 - 2026-04-23 — Исправлен loop при /service/lock**

Раньше при `isLocked` добавлялся только wildcard `/${serviceRoot}/*` который редиректил на `/${serviceRoot}/lock`, создавая loop.

Теперь:
1. Конкретный route `/${serviceRoot}/lock` рендерит `<Layout><LockPage serviceName={manifest.name} /></Layout>`
2. Wildcard `/${serviceRoot}/*` редиректит остальные пути на `/${serviceRoot}/lock`

**Файлы:** `apps/pulse/frontend/src/App.tsx`

**Доработка #10 - 2026-04-23 — ProtectedRoute для /service/lock**

Добавлен `ProtectedRoute` для `/service/lock` чтобы избежать ререндеринга приложения.

**Файлы:** `apps/pulse/frontend/src/App.tsx`
