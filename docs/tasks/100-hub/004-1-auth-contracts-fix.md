# Task 100-hub/004-1: Рефакторинг регистрации (вынос DTO в libs/contracts)

## 🎯 Цель

Перенести контракты (DTO и интерфейсы) для процесса регистрации в общую библиотеку `libs/contracts` и обновить бэкенд и фронтенд для их использования.

## 🧭 Контекст

В рамках задачи 004 была успешно реализована страница регистрации и логика на бэкенде. Однако контракты для запроса регистрации (payload) не были вынесены в `libs/contracts`. В нашей архитектуре монорепозитория любые данные, которые ходят между фронтендом и бэкендом, должны быть строго типизированы через общую библиотеку контрактов. Это гарантирует синхронизацию API.

## 📋 Задачи для выполнения

### 1. Общие контракты (`libs/contracts/src/hub/auth`)
- Создать файл `register.dto.ts` (или добавить в существующий файл с DTO авторизации).
- Описать класс `RegisterDto` (или `RegisterRequestDto`), который должен содержать поля:
  - `username` (string)
  - `email` (string, email)
  - `password` (string, min length 6)
- Добавить необходимые декораторы валидации (`class-validator`, `class-transformer`), если они используются в контрактах, либо описать строгий интерфейс, если классы DTO лежат на бэкенде (предпочтительно держать классы с декораторами прямо в `contracts`, чтобы фронтенд мог переиспользовать типы).
- Убедиться, что `RegisterDto` экспортируется наружу через `libs/contracts/src/hub/auth/index.ts` и `libs/contracts/src/index.ts`.

### 2. Backend (NestJS - `apps/hub/backend`)
- Удалить любые локальные интерфейсы или классы для регистрации (например, если был создан локальный `RegisterDto` внутри `src/auth/dto`).
- Обновить `AuthController` (`@Body() dto: RegisterDto`) и `AuthService`, чтобы они импортировали `RegisterDto` строго из `@app/contracts/hub/auth`.

### 3. Frontend (React - `apps/hub/frontend`)
- Обновить API-вызовы для регистрации (например, в `auth.api.ts` или прямо в `RegisterPage.tsx`), чтобы типизация payload'а (тела запроса) бралась из `@app/contracts/hub/auth`.
- Убедиться, что фронтенд не использует `any` при отправке данных формы на сервер.

## 🛠 Требования и ограничения
- **Strict Mode:** TypeScript в строгом режиме.
- Никаких дублирующихся DTO: бэкенд и фронтенд должны ссылаться на один и тот же импорт из `@app/contracts`.

## 🏁 Критерии приемки (DoD)
1. `RegisterDto` находится в `libs/contracts/src/hub/auth`.
2. Бэкенд контроллер использует этот DTO для валидации входящего запроса.
3. Фронтенд использует этот же DTO (или его тип) для типизации отправляемых данных.
4. Команды `yarn typecheck` проходят успешно в обоих проектах.
5. Написан отчет в формате `004-1-auth-contracts-fix.report.[YYYYMMDD-HHMM].md`.
