# Task 100-hub/003: Реализация аутентификации и управления пользователями

## 🎯 Цель

Реализовать полноценную систему аутентификации на базе JWT в сервисе Hub. Сервис должен уметь регистрировать новых пользователей, аутентифицировать их, выдавать JWT токены и защищать приватные маршруты с помощью Guards.

## 🧭 Контекст

В рамках задач `001-init` и `002-run-and-migrations` мы уже создали базовую сущность `User`, настроили подключение к MySQL и систему миграций. Теперь необходимо оживить эту структуру, добавив бизнес-логику безопасности. Hub — это центральный узел, поэтому его система авторизации будет использоваться как эталонная (в будущем через API Gateway или проверку токенов).

## 📋 Задачи для выполнения

### 1. Backend (NestJS - `apps/hub/backend`)
- Установить необходимые зависимости для работы с JWT и хешированием: `@nestjs/jwt`, `@nestjs/passport`, `passport`, `passport-jwt`, `bcrypt` (и их типы).
- Реализовать `AuthService`:
  - Метод `register` (создание пользователя с хешированием пароля через `bcrypt`).
  - Метод `validateUser` (проверка email и пароля).
  - Метод `login` (генерация JWT токена).
- Реализовать `AuthController`:
  - Эндпойнт `POST /api/auth/register` (использовать DTO, если еще не созданы).
  - Эндпойнт `POST /api/auth/login` (возвращает токен).
- Создать `JwtAuthGuard` и `JwtStrategy` для защиты маршрутов.
- Создать тестовый защищенный эндпойнт `GET /api/auth/me`, возвращающий данные текущего пользователя.

### 2. Общие контракты (`libs/contracts/src/hub/auth`)
- Убедиться, что существуют и актуальны DTO для регистрации (`RegisterRequestDto`) и ответа с токеном (`AuthResponseDto`). При необходимости — добавить или обновить.

### 3. Frontend (React - `apps/hub/frontend`)
- Создать простой сервис `auth.api.ts` (или использовать `axios`/`fetch`) для вызова эндпоинтов `/api/auth/login` и `/api/auth/register`.
- Реализовать простую форму логина на главной странице, которая при успешном входе сохраняет JWT токен в `localStorage` и отображает сообщение об успехе (или данные пользователя).

## 🛠 Требования и ограничения
- **Strict Mode:** Весь TypeScript код должен быть строго типизирован. Использование `any` запрещено.
- **Пароли:** Ни в коем случае не хранить пароли в открытом виде. Использовать `bcrypt` (salt rounds: 10).
- **Секреты:** Секретный ключ для JWT должен браться из переменных окружения (`process.env.JWT_SECRET`), с фолбэком на дефолтное значение для локальной разработки.
- **Миграции:** Если потребуется изменить сущность `User` (например, добавить новые поля), необходимо сгенерировать новую миграцию через `yarn typeorm migration:generate`, а не использовать `synchronize: true`.

## 🏁 Критерии приемки (DoD)
1. Backend успешно компилируется и запускается.
2. Можно зарегистрировать нового пользователя через API (пароль в БД захеширован).
3. Можно войти под созданным пользователем и получить JWT токен.
4. Запрос к защищенному маршруту без токена возвращает `401 Unauthorized`.
5. Запрос с валидным токеном в заголовке `Authorization: Bearer <token>` успешно проходит.
6. Frontend умеет отправлять запросы на логин и обрабатывать ответ.
7. Написан и сохранен отчет `003-auth-implementation.report.md`.
