# Task 100-hub/002: Запуск инфраструктуры Hub и настройка миграций

## 🧭 Контекст

В рамках предыдущей задачи (`001-init`) был написан базовый код для сервиса Hub: созданы NestJS-бэкенд с сущностью `User`, React-фронтенд и обновлен `docker-compose.yml`. Согласно нашим архитектурным правилам, `synchronize: true` в TypeORM запрещен. Поэтому перед полноценным запуском сервисов необходимо настроить механизм миграций, сгенерировать первую миграцию для таблицы пользователей и убедиться, что все Docker-контейнеры работают корректно.

## 🎯 Цели

1. Запустить базу данных `hub-db` через Docker Compose.
2. Настроить конфигурацию TypeORM (Data Source) для запуска миграций через CLI в `apps/hub/backend`.
3. Сгенерировать и применить первую миграцию для сущности `User`.
4. Запустить `hub-backend` и `hub-frontend` через Docker Compose.
5. Проверить работоспособность связки (БД <-> Бэкенд <-> Фронтенд).

## 🛠 Технические требования

- **Миграции:** Использовать TypeORM CLI. Скрипты для генерации и запуска миграций должны быть добавлены в `apps/hub/backend/package.json`.
- **Изоляция БД:** Миграции применяются только к базе данных `hub_db`.
- **Запреты:** ЗАПРЕЩЕНО вручную изменять `.lock` файлы. ЗАПРЕЩЕНО включать `synchronize: true`.

## 📋 План выполнения (Step-by-Step)

### Шаг 1: Запуск базы данных

- Выполни команду для поднятия только базы данных: `docker-compose up -d hub-db`.
- Убедись, что контейнер запустился и MySQL готов к принятию подключений (порт 3306 внутри, 33061 снаружи).

### Шаг 2: Настройка TypeORM CLI

- В `apps/hub/backend/src/` создай файл `data-source.ts` (или `ormconfig.ts`), который будет экспортировать экземпляр `DataSource` для TypeORM CLI. Он должен читать те же переменные окружения, что и `app.module.ts`.
- Добавь директорию для миграций: `apps/hub/backend/src/migrations/`.
- В `apps/hub/backend/package.json` добавь скрипты:
  - `"typeorm"`: вызов CLI TypeORM.
  - `"migration:generate"`: команда для генерации миграции на основе изменений в сущностях.
  - `"migration:run"`: команда для применения миграций к БД.

### Шаг 3: Генерация и применение миграции

- Выполни команду генерации миграции (например, `InitUsers`). Убедись, что файл миграции успешно создался в папке `migrations`.
- Выполни команду применения миграции (`migration:run`).
- Проверь (через логи или подключение к БД), что таблица `user` успешно создана.

### Шаг 4: Запуск и проверка сервисов

- Запусти оставшиеся контейнеры: `docker-compose up -d hub-backend hub-frontend`.
- Проверь логи бэкенда (`docker-compose logs hub-backend`), убедись, что NestJS стартовал без ошибок и подключился к БД.
- Проверь логи фронтенда (`docker-compose logs hub-frontend`), убедись, что Vite сервер работает.
- (Опционально) Сделай тестовый curl-запрос к `http://localhost:3000/api` (или к эндпоинту healthcheck, если он есть), чтобы подтвердить доступность API.

### Шаг 5: Отчетность

- Создай файл отчета `docs/tasks/100-hub/002-run-and-migrations.report.md` с описанием выполненных шагов, созданных скриптов и статусом запуска контейнеров.
- Обнови `docs/04-current-state.md`, указав, что миграции настроены и сервисы Hub успешно запущены.

## ✅ Ожидаемый результат

TypeORM CLI полностью настроен. В базе данных `hub_db` создана таблица `user` через миграцию. Контейнеры `hub-db`, `hub-backend` и `hub-frontend` работают стабильно, не перезагружаются и не выдают ошибок в логах.
