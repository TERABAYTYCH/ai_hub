# Task 100-hub/004-2: Контракты для сущности User и починка ESLint

## 🎯 Цель

Вынести интерфейс/DTO сущности `User` в общую библиотеку контрактов, обновить `LoginResponseDto` для возврата данных пользователя и исправить конфигурацию ESLint в монорепозитории.

## 🧭 Контекст

В рамках задачи 004 была создана сущность пользователя (с ролями, email, username и т.д.). Однако фронтенд не знает структуру этой сущности, так как она не описана в `libs/contracts`. Кроме того, в предыдущем отчете была замечена ошибка конфигурации ESLint, из-за которой не работает линтер. Нам нужно поддерживать чистоту кода и строгую типизацию всех общих сущностей.

## 📋 Задачи для выполнения

### 1. Общие контракты (`libs/contracts/src/hub/users` или `auth`)
- Создать интерфейс `IUser` (или класс `UserDto`), описывающий публичные данные пользователя:
  - `id` (string/uuid)
  - `username` (string)
  - `email` (string)
  - `role` (string - например, 'admin' | 'user')
  - `createdAt` (Date/string)
  - **Важно:** Пароль (`password` или `hash`) здесь быть НЕ ДОЛЖЕН!
- Обновить `LoginResponseDto` в контрактах: он должен возвращать не только `accessToken` (и `refreshToken`, если есть), но и объект `user: IUser`.
- Экспортировать новые типы через `index.ts`.

### 2. Backend (NestJS - `apps/hub/backend`)
- Обновить логику логина (`AuthService.login`), чтобы в ответе, помимо токена, возвращался объект пользователя (без пароля), соответствующий интерфейсу `IUser`.
- Убедиться, что TypeORM-сущность `User` корректно мапится в этот DTO/интерфейс при отправке на клиент.

### 3. Frontend (React - `apps/hub/frontend`)
- Обновить обработку ответа при логине/регистрации: теперь мы получаем не только токен, но и данные пользователя.
- Сохранять данные пользователя (например, в `localStorage` или в стейт), чтобы в будущем отображать его имя/роль в UI (например, в Navbar).

### 4. Исправление ESLint
- Исправить ошибку `неверный путь к tsconfig.base.json`, о которой сообщалось в прошлом отчете.
- Проверить конфигурационные файлы `.eslintrc.js` (или `.eslintrc.json`) в корне и в проектах (`apps/hub/backend`, `apps/hub/frontend`), чтобы путь к `tsconfig` указывал правильно (обычно это `../../tsconfig.base.json` или аналогичный относительный путь).
- Убедиться, что команда `yarn lint` (или `npm run lint`) работает без ошибок конфигурации.

## 🛠 Требования и ограничения
- **Strict Mode:** TypeScript в строгом режиме.
- **Безопасность:** Никогда не передавать хеш пароля на фронтенд.

## 🏁 Критерии приемки (DoD)
1. Сущность `IUser` (или `UserDto`) добавлена в `libs/contracts`.
2. Бэкенд возвращает объект пользователя при успешном логине.
3. Фронтенд типизирует полученного пользователя через контракт.
4. Ошибка конфигурации ESLint исправлена, линтер запускается корректно.
5. Написан отчет в формате `004-2-user-entity-contracts.report.[YYYYMMDD-HHMM].md`.
