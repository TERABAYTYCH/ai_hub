# Task 100-hub/004-3: Строгая привязка сущностей к контрактам (implements)

## 🎯 Цель

Обеспечить строгую проверку типов во время компиляции, обязав backend-сущности и DTO явно реализовывать (implements) общие интерфейсы из `libs/contracts`.

## 🧭 Контекст

В предыдущей задаче мы вынесли интерфейс `IUser` в общую библиотеку. Однако, если TypeORM-сущность `User` на бэкенде не связана с ним через ключевое слово `implements`, TypeScript не сможет гарантировать, что сущность в БД всегда соответствует контракту, который ожидает фронтенд. Явная реализация интерфейсов защитит нас от рассинхронизации данных при будущих изменениях.

## 📋 Задачи для выполнения

### 1. Backend (NestJS - `apps/hub/backend`)
- Открыть файл сущности пользователя (TypeORM entity `User`).
- Добавить явную реализацию интерфейса: `export class User implements IUser { ... }`.
- Проверить и исправить любые ошибки типизации, которые могут возникнуть. 
  *Примечание:* TypeORM обычно использует объекты `Date` для полей `createdAt` и `updatedAt`. Убедитесь, что в интерфейсе `IUser` эти поля допускают тип `Date` (например, `Date | string`), чтобы избежать конфликтов между сериализованным JSON на фронтенде и объектом Date на бэкенде.
- Проверить другие DTO (например, если логин/регистрация используют локальные классы с декораторами валидации, они должны делать `implements` соответствующих интерфейсов запросов из контрактов). Если же DTO импортируются напрямую как классы из `libs/contracts`, то дополнительный `implements` не нужен.

### 2. Проверка сборки
- Убедиться, что маппинг сущности (например, `mapUserToIUser`) работает корректно и не содержит избыточных приведений типов (`as IUser`), если сама сущность уже реализует этот интерфейс.
- Запустить `yarn typecheck` (или `npx tsc --noEmit`) в бэкенде и убедиться, что нет ошибок.

## 🛠 Требования и ограничения
- **Strict Mode:** Никаких `// @ts-ignore` или `any` для обхода несовпадений типов. Если типы не совпадают, нужно исправить либо контракт, либо сущность.

## 🏁 Критерии приемки (DoD)
1. TypeORM сущность `User` содержит `implements IUser`.
2. В случае несовпадения полей (например, при добавлении нового поля в `IUser`) компилятор TypeScript выдает ошибку в классе `User`.
3. Команда `npx tsc --noEmit` в `apps/hub/backend` завершается без ошибок.
4. Написан отчет в формате `004-3-implements-contracts.report.[YYYYMMDD-HHMM].md`.
