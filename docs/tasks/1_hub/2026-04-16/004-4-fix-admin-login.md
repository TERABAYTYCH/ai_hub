# Task 100-hub/004-4: Debug & Fix Admin Login

## 🎯 Цель

Найти причину ошибки авторизации (Login Failed) для сидированного аккаунта администратора и исправить её, чтобы разработчик мог успешно войти в систему.

## 🧭 Контекст

В задаче 004 был добавлен механизм автоматического создания администратора (seeder) при старте приложения с использованием переменных окружения `ADMIN_EMAIL` и `ADMIN_PASSWORD`. Однако при попытке входа с этими данными (или с логином `admin` и паролем `admin`) возникает ошибка авторизации. Вероятно, есть рассинхронизация между тем, как создается пользователь, какие поля ожидает `LocalStrategy` / `AuthService.validateUser`, и тем, что отправляет фронтенд.

## 📋 Задачи для выполнения

### 1. Инспекция базы данных и сидера
- Проверить, создается ли вообще пользователь в базе данных при старте бэкенда.
- Проверить код сидера (`OnApplicationBootstrap`): 
  - Корректно ли хешируется пароль перед сохранением (используется ли `bcrypt.hash`)?
  - Какие поля заполняются при создании? Заполняется ли `username` (например, значением `admin`) или только `email`?

### 2. Инспекция процесса логина (Backend)
- Проверить `LocalStrategy` и `AuthService.validateUser`: по какому полю происходит поиск пользователя в БД? По `username` или по `email`?
- Проверить `LoginRequestDto`: какие поля он требует?
- Убедиться, что `bcrypt.compare` корректно сравнивает введенный пароль с хешем из БД.

### 3. Инспекция Frontend
- Проверить форму на `LoginPage.tsx`: какое поле отправляется на сервер в качестве логина (`username` или `email`)?
- Убедиться, что отправляемые данные точно соответствуют `LoginRequestDto`.

### 4. Исправление
- Привести весь флоу к единому стандарту. Рекомендуемый подход: 
  - Пользователь может входить по `username`.
  - Сидер должен создавать пользователя с `username: 'admin'`, `email: process.env.ADMIN_EMAIL` и паролем из `process.env.ADMIN_PASSWORD`.
  - Фронтенд должен отправлять `username` и `password`.
- Если бэкенд не был перезапущен после добавления `.env`, убедиться, что переменные окружения загружаются корректно (например, через `ConfigModule.forRoot()`).

## 🏁 Критерии приемки (DoD)
1. Найдена и описана в отчете точная причина ошибки логина.
2. Внесены исправления в код (backend/frontend).
3. Можно успешно зайти на фронтенде, используя логин `admin` и пароль `admin` (или те, что указаны в `.env`).
4. Написан отчет в формате `004-4-fix-admin-login.report.[YYYYMMDD-HHMM].md`.
