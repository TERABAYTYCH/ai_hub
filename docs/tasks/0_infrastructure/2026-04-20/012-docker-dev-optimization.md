# Task: Оптимизация Docker-окружения и переход на Dev-режим (Hot-Reload)

Постоянная пересборка Docker-контейнеров (`docker compose build`) занимает слишком много времени. Нам нужно перевести локальную разработку в режим `watch/dev`, сохранив при этом работоспособность Module Federation (доступ к `remoteEntry.js`).

Твоя задача — изменить конфигурацию запуска фронтенд-приложений так, чтобы при старте контейнера автоматически устанавливались зависимости, собирался актуальный `remoteEntry.js`, а затем запускался dev-сервер Vite с поддержкой hot-reload.

---

## 🛠 Что нужно сделать (Пошаговый план)

### 1. Обновление скриптов запуска (`package.json`)

В `apps/hub/frontend/package.json` и `apps/pulse/frontend/package.json` (и других микрофронтендах, если есть) добавь новый скрипт для Docker:

- `"dev:docker": "yarn install && yarn build && vite"`
  _Логика:_ При каждом старте контейнера мы гарантируем актуальность пакетов и собираем свежий билд (чтобы сгенерировать `dist/assets/remoteEntry.js`), после чего поднимаем быстрый dev-сервер.

### 2. Настройка Docker Compose (`docker-compose.yml`)

Измени конфигурацию фронтенд-сервисов:

- Убедись, что исходный код прокинут через `volumes` (например, `- ./apps/hub/frontend:/app`).
- Замени команду запуска (или `entrypoint`) на вызов нового скрипта: `command: yarn dev:docker`.
- Убедись, что папка `node_modules` и `dist` вынесены в анонимные тома (чтобы они не перезаписывались с хоста, если ты используешь локальный маппинг).

### 3. Настройка Vite (`vite.config.ts`)

Dev-сервер Vite по умолчанию собирает файлы в памяти и не отдает физический `remoteEntry.js` из папки `dist`. Напиши простой локальный плагин внутри `vite.config.ts` с использованием `configureServer`:

- Плагин должен перехватывать запросы к `/assets/remoteEntry.js` (или другим файлам федерации).
- Если запрос совпадает, плагин должен читать файл из локальной директории `dist/assets/` и отдавать его с правильными заголовками (CORS, Content-Type: application/javascript).

### 4. Обновление документации (`03-execution-pipeline.md`)

Перепиши файл `docs/tasks/.../03-execution-pipeline.md` (или где он у нас лежит), актуализировав процесс разработки.

| **Шаг**                        | **Было (Старый пайплайн)**                    | **Стало (Новый пайплайн)**                |
| ------------------------------ | --------------------------------------------- | ----------------------------------------- |
| **При изменении кода**         | Остановка -> `docker compose build` -> Запуск | Код обновляется мгновенно (Vite HMR)      |
| **При изменении зависимостей** | Полная пересборка образа                      | `docker compose restart <service>`        |
| **Доступ к remoteEntry**       | Из статического билда Nginx/Serve             | Перехватывается Vite и отдается из `dist` |

В документе четко укажи: **"Вместо пересборки проекта теперь достаточно просто перезапустить контейнеры (`docker compose restart`), так как установка пакетов и сборка происходят при старте контейнера."**

## ⚠️ Критические требования

1. **Проверка типов и линтеры:** Перед завершением задачи обязательно запусти `yarn typecheck` и `yarn lint`. Убедись, что изменения в `vite.config.ts` не ломают типизацию.
2. **Работоспособность HMR:** Убедись, что проброс портов и томов в Docker настроен так, что Hot Module Replacement (HMR) работает корректно при изменении файлов на хосте.
