# Task 009: Настройка Nginx API Gateway и маршрутизации микросервисов

**Домен:** `0_infrastructure`
**Статус:** К выполнению
**Ответственный:** PM / Tech Lead (Оркестратор)

## 1. Бизнес-контекст

Текущая инфраструктура запущена, но доступ к сервисам осуществляется напрямую через разные порты.
**Твоя цель:** Настроить Nginx в качестве единого API Gateway на порту 80 (`localhost`), чтобы все микросервисы были доступны через единую точку входа. Структура URL должна быть сгруппирована по имени сервиса.

---

## 2. Архитектурные требования и правила маршрутизации (CRITICAL)

Тебе необходимо спроектировать и реализовать обновление конфигурации `nginx/nginx.conf` (и связанных файлов) по следующим правилам:

### 2.1. Маршрутизация Backend (API) — Высокий приоритет

Все API-запросы должны содержать `/api/` **после** имени сервиса и проксироваться на соответствующие NestJS бэкенды:

- `localhost/hub/api/*` ➔ `hub-backend:3000`
- `localhost/pulse/api/*` ➔ `pulse-backend:3001`
- `localhost/service/api/*` ➔ `service-backend:3002`
- `localhost/control/api/*` ➔ `control-backend:3003`

### 2.2. Маршрутизация Frontend (UI) — Низкий приоритет

Запросы в браузере (не содержащие `/api/`) должны отдавать статику соответствующих React-приложений:

- `localhost/` и `localhost/hub/*` ➔ `hub-frontend` (порт 5173)
- `localhost/pulse/*` ➔ `pulse-frontend` (порт 5174)
- `localhost/service/*` ➔ `service-frontend` (порт 5175)
- `localhost/control/*` ➔ `control-frontend` (порт 5176)

_Важно: Разработчику потребуется правильно настроить порядок `location` в Nginx (API должно перехватываться раньше, чем UI), а также обновить параметр `base` в `vite.config.ts` каждого фронтенда для корректной работы по подпутям._

### 2.3. Инфраструктура

- Nginx уже запущен в `docker-compose.yml` на 80 порту.
- Конфигурация должна поддерживать WebSocket-соединения (для Vite HMR в режиме разработки).
- Настройки CORS в Nginx или NestJS должны разрешать запросы внутри единого домена.

## 3. Жесткие критерии приемки (Acceptance Criteria & QA)

Ты **ОБЯЗАН** убедиться, что субагенты выполнили следующие проверки через терминал, прежде чем рапортовать о завершении задачи:

1. **Frontend роутинг:** Вызов `curl -I http://localhost/pulse/` должен возвращать `HTTP/1.1 200 OK` (отдача HTML фронтенда).
2. **API роутинг:** Вызов `curl -I http://localhost/hub/api/` (или любого доступного эндпоинта, например `/hub/api/health`) должен доходить до NestJS и возвращать корректный статус (например, 200, 401 или 404 от самого NestJS, но **не 502 Bad Gateway** от Nginx).
3. **Изоляция:** Запрос `http://localhost/pulse/api/` не должен возвращать HTML-страницу фронтенда. Он должен строго уходить на бэкенд.
4. **Vite HMR (WebSockets):** В логах Nginx (`docker compose logs nginx`) не должно быть ошибок `Connection refused` или `Upgrade request failed` при попытке установить WebSocket соединение.
5. **Загрузка статики:** При запросе страниц статика (JS/CSS файлы) должна загружаться без ошибок 404.
