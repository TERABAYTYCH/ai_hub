# Задача 008: Стандартизация логики авторизации и роутинга в libs/ui-kit

## Контекст
После вынесения UI-компонентов авторизации в `libs/ui-kit` обнаружилось, что поведение приложений (Hub и Pulse) различается. Например, в Pulse кнопка Logout не делает редирект на `/login`, а авторизованный пользователь может случайно зайти на страницу `/login` без автоматического редиректа на главную.
Чтобы обеспечить одинаковый UX во всех микрофронтендах, необходимо вынести базовую логику управления сессией (токены, login, logout) и компоненты защиты маршрутов в общую библиотеку.

**ВНИМАНИЕ:** При написании React-кода обязательно использовать скилл `vercel-react-best-practices`.

## Требования и шаги реализации

### 1. Создание AuthContext в ui-kit
- Создать `AuthContext` и провайдер `AuthProvider` в `libs/ui-kit/src/providers/AuthProvider.tsx`.
- Провайдер должен инкапсулировать логику работы с `localStorage` (чтение/запись токенов).
- Провайдер должен предоставлять состояние `isAuthenticated`, данные пользователя (если есть), а также методы `login(token)` и `logout()`.
- Метод `logout()` должен очищать токены и делать принудительный редирект на `/login` (используя `useNavigate` из `react-router-dom`).

### 2. Создание компонентов-оберток для роутинга в ui-kit
- Создать `ProtectedRoute.tsx` в `libs/ui-kit/src/components/auth/`:
  - Если пользователь НЕ авторизован (`!isAuthenticated`), компонент должен делать редирект (`<Navigate to="/login" replace />`).
  - Иначе рендерить `children` (или `Outlet`).
- Создать `GuestRoute.tsx` (или `PublicRoute.tsx`):
  - Если пользователь УЖЕ авторизован, делать редирект на главную (`<Navigate to="/" replace />`).
  - Иначе рендерить `children` (формы логина/регистрации).

### 3. Экспорт из ui-kit
- Добавить экспорты `AuthProvider`, `useAuth` (хук для контекста), `ProtectedRoute` и `GuestRoute` в `libs/ui-kit/src/index.ts`.

### 4. Рефакторинг Hub Frontend
- Обернуть корневой компонент приложения в `AuthProvider`.
- Заменить локальные реализации защищенных маршрутов на `ProtectedRoute` и `GuestRoute` из `ui-kit`.
- Обновить логику в `Header` (или где вызывается логаут) на использование `logout` из `useAuth()`.
- Убедиться, что при успешном логине вызывается `login(token)` из контекста.

### 5. Рефакторинг Pulse Frontend
- Проделать те же шаги, что и для Hub Frontend: обернуть в `AuthProvider`, использовать `ProtectedRoute` и `GuestRoute`.
- Удалить дублирующуюся логику работы с токенами из `apps/pulse/frontend/src/utils/auth.ts` (если она полностью покрывается контекстом, либо оставить только утилиты для Axios-интерцепторов).
- Убедиться, что кнопка Logout в Pulse теперь корректно очищает сессию и перебрасывает на `/login`.

## Критерии приемки (DoD)
1. Логика авторизации (провайдер и роуты) находится в `libs/ui-kit`.
2. В Hub и Pulse при клике на Logout происходит очистка токенов и редирект на `/login`.
3. В Hub и Pulse авторизованный пользователь при попытке зайти на `/login` или `/register` автоматически перенаправляется на `/`.
4. В Hub и Pulse неавторизованный пользователь при попытке зайти на защищенные страницы перенаправляется на `/login`.
5. Сборка всех проектов (`turbo build`) проходит без ошибок.
